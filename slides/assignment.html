Make the workflow yours, and make it the gatekeeper

In the We-do you set the shipped answer aside — git mv .github/workflows/ci.yml .github/ci-reference.yml — and rebuilt the two-job skeleton below from scratch (full version in exercises/lab.md). Pick up from there:

name: CI
on: [pull_request]            # ← you'll refine this
permissions: {contents: read}
jobs:
  validate:  # checkout → scripts/validate.sh
  lint:      # checkout → setup-python "3.12" → pip install
             #   yamllint · actionlint · shellcheck · ign-lint
  • Steps 1-2: path filters + compose validation.
  • Steps 3-4: status badge + branch protection on main: direct pushes rejected, red PRs blocked.
  • Step 5: sanity check against the shipped .github/workflows/ci.yml.

PART 2 · 1-2
Path filters & compose validation
Don't spin up two VMs to lint files a PR never touched.
1
Add the paths: filter (see block) plus a push: branches: [main] backstop trigger.
1
Prove it: open a PR that touches only README.md → the workflow must be skipped, not run-and-passed. Check the PR's checks panel. Keep that PR open — it springs a trap in step 4.
2
Add docker compose config -q as a final step of the lint job. It catches Compose-level issues yamllint can't see: undefined services, port-string typos, malformed env maps.
on:
  pull_request:
    paths:
      - "projects/**"
      - "scripts/**"
      - "docker-compose.yml"
      - ".github/workflows/**"
      - ".yamllint.yml"
      - "rule_config.json"
  push:
    branches: [main]

# lint job, final step:
      - run: docker compose config -q
Filter discipline

Too narrow → view changes sneak past CI. Too broad ("**") → the filter is pointless. Include the configs the checks read, rule_config.json too.

3

Try the forbidden route: commit straight to main and push (see block). GitHub rejects the push. Direct commits to main are dead.
4

4

# 4: the rule on main (Settings → Branches)
[✓] Require a pull request before merging  # no direct commits
[✓] Require status checks to pass          # select lint + validate
[✓] Block force pushes                     # no rewriting history
[✓] Restrict deletions                     # main can't be deleted

Self-hosted runners: watch, then argue

Instructor demo: an ephemeral Docker runner registers against a repo, one job routes to it with runs-on: [self-hosted, local-lab03], and you watch it execute in docker logs. Hands-on comes in Labs 04 and 05.

  • Watch for: the short-lived registration token (via gh api), the runner appearing in Settings → Actions → Runners, "Listening for Jobs" in the logs, and the runner terminating itself after one job (EPHEMERAL=true).
  • Watch for the gotcha: the runner only has what you put on it, with no preinstalled shellcheck. "Works on ubuntu-latest" is an assumption, not a law.
  • Discussion, come with an answer: which of your deploys would need a self-hosted runner, and why specifically? ("Our gateway is behind the customer's VPN; GitHub-hosted physically can't reach it" beats "maybe.")

Stretch 0: register a runner, run a job on your own machine
You watched the demo. Now do it in your fork: an ephemeral Docker runner claims one job and terminates. Full how-to in docs/self-hosted-runners.md.
1
Mint a short-lived token and start the runner (see block). It registers against your fork, prints "Listening for Jobs", and appears in Settings → Actions → Runners. Needs Docker running.
2
Dispatch the job. The repo ships .github/workflows/self-hosted-demo.yml (manual-only, runs-on: [self-hosted, local-lab03]). Fire it with gh workflow run and watch it execute in docker logs, not on GitHub's infra.
3
Meet the gotcha. The last step probes for shellcheck. Your bare runner doesn't have it: "works on ubuntu-latest" was an assumption. The runner has only what you put on it.
4
Clean up (part of the exercise). Ephemeral + --rm means it self-removes after one job; confirm it's gone from the Runners list. Any PAT you minted: revoke it.
# 1: short-lived registration token (your fork)
export RUNNER_TOKEN="$(gh api -X POST \
  repos/<you>/cicd-lab-03-github-actions/actions/runners/registration-token \
  --jq .token)"

docker run -d --rm --name lab03-runner \
  -e REPO_URL="https://github.com/<you>/cicd-lab-03-github-actions" \
  -e RUNNER_TOKEN="$RUNNER_TOKEN" \
  -e LABELS="self-hosted,local-lab03" \
  -e EPHEMERAL=true \
  -v /var/run/docker.sock:/var/run/docker.sock \
  myoung34/github-runner:latest

docker logs -f lab03-runner   # wait for "Listening for Jobs"

# 2: dispatch the job at your local runner
gh workflow run self-hosted-demo.yml
# → the job runs in docker logs, then the runner exits
The one rule

A self-hosted runner runs arbitrary workflow code on a machine inside your network. Never attach one to a public repo that takes fork PRs: a malicious PR runs on your box. Ephemeral runners, trusted repos.

Two nets

Pre-commit is the fast, local net; CI is the enforced, shared one. You want both: local for speed, CI because hooks are opt-in and can be skipped.

Stretch 2: three small workflow extras
Independent extras from exercises/lab.md: do any, all, or none.
A
Matrix ign-lint over individual views so each view surfaces as its own check. One entry today, but it's the pattern that scales as the HMI grows: it isolates which view broke, not speed.
B
Cancel superseded runs. Add a workflow-level concurrency group keyed on the ref so a fresh push cancels the in-flight run on the same branch, instead of both burning minutes to the end.
C
Pin actions by SHA. Swap one @v4 tag for its full commit SHA. A tag can be silently re-pointed by its owner: a supply-chain risk that bites harder once a runner sits inside a plant network. Let Dependabot bump it.
# A: the matrix pattern
  lint-views:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        view:
          - "projects/lab-project/com.inductiveautomation.perspective/views/pages/overview/view.json"
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: {python-version: "3.12"}
      - run: pip install ign-lint==0.6.1
      - run: ign-lint --config rule_config.json --files "${{ matrix.view }}"

# B: cancel superseded runs (workflow-level)
concurrency:
  group: "${{ github.workflow }}-${{ github.ref }}"
  cancel-in-progress: true

# C: pin an action by SHA, not a movable tag
      - uses: actions/checkout@08c6903  # v4.x
Why they help

Each is a real hardening habit in miniature: named checks that point at the broken view, no wasted minutes on stale runs, and a tag that can't be re-pointed under you.
